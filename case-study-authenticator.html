Overview

Where this started

PPFAS Partners is a web app used by ARN-holding mutual fund distributors — people managing client portfolios and processing transactions on behalf of real families, so security is not something you take lightly. The platform already had 2FA through SMS and email OTP, and it worked fine for most people.

But a few distributors started bringing up the same thing — not as a complaint, more like a question. They were already using Google Authenticator or Authy for other apps and found it faster. Some of them work in areas where mobile network isn't reliable. Waiting on an OTP that may or may not arrive — while someone's money is sitting in a pending transaction — is not a great experience.

Before building, business, tech and stakeholders discussed feasibility and value. We also looked at how top financial apps handle this — most offer authenticator as an additional method, not a replacement. That made the direction clearer. My focus from there was entirely on the user side: making setup feel non-technical, and making sure people who never asked for this wouldn't notice anything changed.

Method selection — Authenticator · SMS · Email
Challenge

The feature was simple. The users weren't power users.

From a technical side, TOTP is a well-understood problem. From a design side, the question was harder: how do you guide someone through downloading an app, scanning a QR code, and entering a 6-digit code that refreshes every 30 seconds — without making them feel lost?

These distributors log in maybe twice a week. Some would get it immediately; others had never seen a QR setup flow before. Both types of users had to end up in the same place — set up and confident.

One thing I was firm about from the start: enabling authenticator should never silently remove the old options. A distributor locked out on a trading day because they couldn't find their phone — that's not a UX mistake, that's a trust problem. Switching had to feel optional and reversible, always.

Strategy

I mapped the flows before opening Figma

There are two separate journeys here — a one-time setup, and the everyday login after that. I mapped both to understand where someone might drop off or get confused before designing anything.

Setup flow — happens once Manage 2FA settings → Select authenticator method → Download app prompt → Scan QR code → Verify & activate
Login flow — every session after setup ARN + password → Select 2FA method → Enter TOTP code → Access granted

Two likely drop-off points stood out immediately — the moment between "select method" and "download the app," and the QR scan step itself. Those two became the focus of every iteration.

Three versions before it felt right

The first version wasn't bad — it just had gaps I didn't see until I looked at it from a user's perspective. Same with the second. Each round made the next one easier to get right.

Problem

The QR code appeared with nothing telling the user what to do before they got there. If they hadn't downloaded an authenticator app yet, they were just stuck — staring at a code with no context. That's where people would drop off.

V2 — Revised

Added a full page for the download step

Screen placeholder — V2
Problem

A separate page broke the flow. Users navigated away to download the app, and when they came back they'd lost track of where they were. Some had to restart entirely. It fixed one problem and created another.

Final — Shipped

Modal overlay keeps the user in context

Screen placeholder — Final
Why this worked

Putting the download guidance in a lightweight modal kept the settings page visible behind it. The user never lost their place. I also added the manual secret key below the QR code so users whose camera didn't cooperate had a fallback — no dead end, no support call needed.

Final design

The final flow, screen by screen

Everything used the existing component library — same spacing, same typography, same patterns. The goal was for this to feel like it had always been there, not something added later.

Manage 2FA screen

Managing 2FA — both methods, same page

Both options sit on the same settings page with clear separation. Neither is pre-selected — the user chooses when they're ready. An "Active" badge shows which method is live, so there's never confusion about their security state. Going back to SMS is always right there.

Download modal

Download prompt — modal, not a new page

The download guidance appears as a modal overlay with the settings page visible behind it. Three app options are listed — Google Authenticator, Authy, Microsoft Authenticator — so it works with whatever the user already has. A quiet note recommends cloud backup, addressing "what if I lose my phone" without making it feel alarming.

QR code screen

QR code — with a fallback built in

The QR code is the primary path, with the manual secret key right below it for anyone whose camera isn't working. Setup completes only after entering both MPIN and the 6-digit code — confirming the app actually works before they're fully enrolled.

Results

What happened after it shipped

The feature launched as a fully optional addition. No one was forced to switch, and no one who didn't care about it noticed anything different. The distributors who did switch got what they wanted — codes that generate instantly, work offline, and don't depend on network availability at login time.

Post-launch feedback was mostly quiet, which was the goal. No major usability issues were raised; a few small copy tweaks based on early feedback, but the core flow held up without structural changes.

From a security standpoint, time-based codes are harder to intercept than SMS OTPs — so this wasn't just a convenience improvement; it was a genuinely better method for a platform handling financial transactions. The settings page is also structured so a future method can slot in alongside what's already there, without a redesign.
